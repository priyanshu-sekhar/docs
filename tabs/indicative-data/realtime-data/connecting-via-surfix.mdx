## Introduction

The Parameta SURFIX™ **Rules of Engagement** document describes the features and functionalities of the Parameta Market Data FIX Gateway (SURFIX).

This document does not cover how to implement the FIX Protocol itself. It covers the requirements for integrating with Parameta using FIX. Use it alongside the Parameta **SURFIX™ Specification** document, which defines the FIX messages and fields supported by SURFIX™.

## Connectivity and setup

<Steps>
  <Step title="Set up a FIX connection">
    As a centralized service, Parameta listens for FIX connection requests from the client. Each client is assigned a unique **SenderCompID (49)** and **TargetCompID (56)** pair for each session.
  </Step>
  <Step title="Configure your user session">
    Clients are allocated a unique company ID, by mutual agreement with Parameta, on a per FIX user session basis. Include the Company ID in **FIX Tag 49 (SenderCompID)** of the Logon (35=A) message.
  </Step>
  <Step title="Complete onboarding and testing">
    All clients are advised to perform testing in the UAT environment before receiving live market data. Refer to the **SURFIX™ Onboarding Document** for recommended test scenarios.
  </Step>
  <Step title="Connect to production">
    Parameta provides a production environment segregated from any testing environment. Contact Parameta for connection details.
  </Step>
</Steps>

### Network connectivity

SURFIX is offered over the internet via an F5 solution. SSL/TLS encryption secures connections between client and Parameta servers.

Cipher selection is determined by negotiation and can be restricted to a specific set of supported protocols and ciphers. The level of supported protocols and ciphers changes with each major F5 software release.

<Note>
  TPICAP currently runs F5 version 12.1.
</Note>

### User-defined messages, components, tags, and values

The SURFIX Specification contains user-defined (custom) messages, components, tags, and values designed to deliver specific data Parameta currently distributes.

<AccordionGroup>
  <Accordion title="Custom tags" icon="tag">
    Tags numbered between **20000 and 39999** are user-defined tags. They are identified by a `*` symbol next to their names in the specification document.
  </Accordion>

  <Accordion title="Custom values" icon="asterisk">
    Some non-standard values for standard tags are adopted. They are identified by a `*` symbol next to their descriptions in the specification document (for example, `269=v` (Volatility)).

    Some standard components and tags may be adopted within standard components and messages. For example, `RelativeValueGrp` is not part of Market Data messages in the FIX standard, but it is used in those messages in SURFIX.
  </Accordion>
</AccordionGroup>

### Market data entry types

Market Data entries are classified by their type using the **MDEntryType (269)** tag. SURFIX supports many of the standard entry types as well as some custom values.

| Custom value | Description |
| --- | --- |
| `269=p` | **Product Specific Data** — Attributes related to the security and not classified as any of the standard entry types. |
| `269=v` | **Volatility** — Volatility data, further categorized by the `VolatilityType (21000)` custom tag. |

### Market data entry identifier

A unique **MDEntryID (Tag 278)** integer value is assigned to each Market Data entry (in `MDFullGrp`) and is used to identify specific entries when sending updates.

<Note>
  This identifier is unique for each client connection and resets to `1` every time a client establishes the FIX connection. When a market data entry is deleted, its MDEntryID is not reused within the same session.
</Note>

`MDEntryID` is used to identify the entry being updated in MarketDataIncrementalRefresh (35=X) messages.

### Updates to market data entries

Data distributed in `MDFullGrp` tags have an `MDEntryType` defined and a unique `MDEntryID` assigned. `MDEntryID` identifies the entry being updated in MarketDataIncrementalRefresh (35=X) messages.

Any updates to tags which are not part of `MDFullGrp` (and therefore have no `MDEntryID`) are included in any of the `MDIncGrp` components.

<Note>
  Net change from the previous day's closing price is sent in the **NetChgPrevDay (451)** tag, which is in the body of the MarketDataSnapshotFullRefresh (35=W) message (not in `MDFullGrp`) but in the `MDIncGrp` component. Updates to this tag are merged with any other updates sent for the security.
</Note>

The type of Market Data Update Action is defined using the **MDUpdateAction (279)** tag:

| Value | Name | Use case |
| --- | --- | --- |
| `279=1` | Change | Updating an existing **non-pricing** data entry (when `MDEntryType` is `3` (Index Value), `B` (Trade Volume), `p` (Product Specific Data), or `v` (Volatility)) with a new value. |
| `279=5` | Overlay | Updating an existing **pricing** data with a new value. Overlay action does not delete unchanged data in the market data entry. |

```text Format

```

The Parameta Security ID is included in the **SecurityID (48)** tag, with **SecurityIDSource (22)** set to `M` (Marketplace-assigned identifier) in the Instrument component.

<Note>
  The Instrument component is included only in the first `MDIncGrp` component in MarketDataIncrementalRefresh (35=X) messages.
</Note>

### Date and time of market data entry

**MDEntryDate (272)** and **MDEntryTime (273)** tags contain the date and time of market data entries and updates.

- `MDEntryTime` and `MDEntryDate` are sent in the first `MDFullGrp` repeat in MarketDataSnapshotFullRefresh (35=W) messages. They are considered valid for all data points within the same message.
- The tags are included in the first `MDIncGrp` component in MarketDataIncrementalRefresh (35=X) messages if their values differ from the previously sent values.
- Date and time information is considered valid for all data points sent in `MDIncGrp` components within the same message.

### Record templates

Different securities in different asset classes contain different data sets. The Record Template concept classifies different data patterns in the Market Data Feed. Each template is represented by an integer value reported within the **MDRecordTemplate (21999)** tag in 35=W and 35=X messages.

<Tip>
  Clients can request the **SURFIX Record Template document** for a detailed description of each Record Template.
</Tip>

### Tenor data

SURFIX uses the following custom tags for tenor data:

| Tag | Field |
| --- | --- |
| `20001` | TenorOne |
| `20002` | TenorTwo |
| `20003` | TenorThree |
| `20004` | TenorFour |

Up-to-date supported values are listed in the **Period List document**, downloadable from the client zone.

### Pricing data

Pricing data is distributed in the **MDEntryPx (270)** tag, further qualified by **PriceType (423)** where possible.

Some records (for example, US Government Bonds) have prices quoted in thirty-seconds and may contain a plus symbol at the end (for example, `98.22+`).

<Note>
  SURFIX handles thirty-seconds pricing by replacing the `+` sign with the number `4` and setting `PriceType(423)=17` (Product ticks in thirty-seconds).
</Note>

## FIX dictionaries

FIX Dictionaries are machine-readable representations of FIX Specifications in XML format. They contain all tags, values, repeating groups, and messages supported by SURFIX.

SURFIX supports multiple dictionaries to enable clients to receive different data sets using different Comp IDs. Instances with multiple dictionaries must be set up by Parameta Technical teams.

<Tip>
  **FIX Antenna®**-compatible dictionaries are made available to clients in the client zone.
</Tip>

## Data feeds

Clients must establish a connection and send requests to receive both pricing and security list data. If a request arrives before data is available, Parameta accepts the request but does not send data until it becomes available.

<Warning>
  During any downtime, available market data is **not cached**. Only current market data is reported upon reconnection.
</Warning>

If the FIX session terminates while a subscription is active, the subscription is considered terminated. Once the FIX session is re-established, a new request must be sent — and the incoming and outgoing Message Sequence Numbers must be reset.

## Market data

### Subscription request and response flow

```text Successful subscription
FIX Client                                                  SURFIX
   |                                                           |
   |  MarketDataRequest (35=V)                                 |
   |  MDReqID (262) = <Client's ID>                            |
   |  SecurityID (48) = [ALL]                                  |
   |  SubscriptionRequestType (263) = 1 (Subscribe)            |
   |  MDUpdateType (265) = 1 (Incremental)                     |
   | --------------------------------------------------------> |
   |                                                           |
   |  MarketDataSnapshotFullRefresh (35=W)                     |
   |  SecurityID (48) = FXSPTGBPUSDSPT:GBL.BIL.QTE.RTM!IC      |
   | <-------------------------------------------------------- |
   |  MarketDataSnapshotFullRefresh (35=W)                     |
   |  SecurityID (48) = FXSPTEURUSDSPT:GBL.BIL.QTE.RTM!IC      |
   | <-------------------------------------------------------- |
   |  MarketDataSnapshotFullRefresh (35=W)                     |
   |  SecurityID (48) = FXSPTUSDTRYSPT:GBL.BIL.QTE.RTM!IC      |
   | <-------------------------------------------------------- |
   |                                                           |
   |  MarketDataIncrementalRefresh (35=X)                      |
   |  MDReqID (262) = <Client's ID>                            |
   | <-------------------------------------------------------- |
```

On successful subscription, SURFIX sends one MarketDataSnapshotFullRefresh (35=W) message for each security the client is permissioned for. MarketDataIncrementalRefresh (35=X) messages continue thereafter.

### Request reject flow

```text Rejected request
FIX Client                                          SURFIX
   |                                                   |
```

## Security list

The Security List functionality enables clients to access the updated list of securities for which they are permissioned to receive market data.

SURFIX sends the list of securities in an order using SecurityList (35=y) messages and maintains the list (and its order) by reporting updates using SecurityListUpdateReport (35=BK) messages.

### Subscription request and response flow

```text Successful subscription
FIX Client                                          SURFIX
   |                                                   |
   |  SecurityListRequest (35=x)                       |
   |  SecurityReqID (320) = <Client's ID>              |
   |  SecurityListRequestType (559) = 4                |
   |  SubscriptionRequestType (263) = 1 (Subscribe)    |
   | ------------------------------------------------> |
   |                                                   |
   |  SecurityList (35=y)                              |
   |  Note: Large Security List messages may be        |
   |  fragmented.                                      |
   | <------------------------------------------------ |
   |                                                   |
   |  SecurityListUpdateReport (35=BK)                 |
   |  MDReqID (262) = <Client's ID>                    |
   |  Updates for Security List                        |
   | <------------------------------------------------ |
```

On successful subscription, SURFIX sends a SecurityList (35=y) message containing the list of securities the client has permission for. SecurityListUpdateReport (35=BK) messages are sent when any of the securities change in a market.

### Request reject flow

```text Rejected request
FIX Client                                          SURFIX
   |                                                   |
   |  SecurityListRequest (35=x)                       |
```

### Unsubscribing

Active Security List requests can be cancelled using a SecurityListRequest (35=x) message that references the original `SecurityReqID (320)` and sets **SubscriptionRequestType (263) = 2**.

SURFIX stops sending security list updates immediately after receiving the disable subscription request.

### Fragmented messages

Security Lists containing a large number of securities may be sent in multiple Security List (fragmented) messages.

When a Security List Message is fragmented, the following additional tags are included:

| Tag | Field | Description |
| --- | --- | --- |
| 393 | TotNoRelatedSym | The total number of securities the chain contains (in all fragments). |
| 893 | LastFragment | `N` for fragments other than the last; `Y` for the last fragment of the chain. |

<Note>
  If the message is not fragmented, these tags are not included.
</Note>
